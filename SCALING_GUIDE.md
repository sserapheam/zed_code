# Руководство по масштабированию zedcode

## 🚨 Критические проблемы безопасности (исправить в первую очередь!)

### 1. Изоляция выполнения кода

**Текущая проблема:** Код выполняется через `subprocess.Popen` без изоляции, что позволяет:
- Удалять файлы на сервере
- Читать системные файлы
- Делать сетевые запросы
- Потреблять неограниченные ресурсы

**Решение:** Использовать Docker контейнеры с ограничениями

## 📋 План действий по масштабированию

### Этап 1: Безопасность (КРИТИЧНО - сделать немедленно)

#### 1.1. Docker-контейнеризация выполнения кода

**Необходимо:**
- Docker Engine
- Docker API для Python (`docker` библиотека)

**Преимущества:**
- Полная изоляция выполнения
- Ограничение ресурсов (CPU, память, диск)
- Автоматическая очистка после выполнения
- Защита от атак

**Пример архитектуры:**
```python
# Новая функция безопасного выполнения
def run_code_in_container(code: str, language: str, timeout: int = 5):
    client = docker.from_env()
    container = client.containers.run(
        image=f"zedcode-{language}:latest",
        command=f"python /app/runner.py",
        stdin_open=True,
        mem_limit="128m",  # Максимум 128MB памяти
        cpu_period=100000,
        cpu_quota=50000,   # 50% одного CPU
        network_disabled=True,  # Отключить сеть
        read_only=True,    # Только чтение файловой системы
        remove=True,       # Удалить после выполнения
        detach=False,
        stdin=code.encode(),
        timeout=timeout
    )
```

#### 1.2. Ограничения ресурсов

**Необходимо установить:**
- **Память:** 64-256MB на выполнение
- **CPU:** 0.5-1 ядро на выполнение
- **Время:** 5-10 секунд максимум
- **Диск:** 50MB максимум

### Этап 2: Асинхронная обработка (для масштабирования)

#### 2.1. Очередь задач (Celery + Redis)

**Зачем:**
- Выполнение кода не блокирует веб-сервер
- Можно обрабатывать несколько задач параллельно
- Автоматическое перераспределение нагрузки

**Настройка:**
```python
# celery_app.py
from celery import Celery

celery_app = Celery(
    'zedcode',
    broker='redis://localhost:6379/0',
    backend='redis://localhost:6379/0'
)

@celery_app.task(bind=True, max_retries=3)
def execute_code_task(self, code, language, tests, user_id):
    try:
        # Выполнение в Docker контейнере
        result = run_code_in_container(code, language)
        return result
    except Exception as exc:
        raise self.retry(exc=exc)
```

**В Flask:**
```python
# Вместо синхронного выполнения
task = execute_code_task.delay(user_code, language, tests, user_id)
return jsonify({"task_id": task.id, "status": "processing"})
```

### Этап 3: Масштабирование базы данных

#### 3.1. Оптимизация PostgreSQL

**Необходимо:**
- Connection pooling (pgBouncer)
- Индексы на часто используемые поля
- Репликация для чтения (1 master + N replicas)

**Настройки для 100-1000 одновременных пользователей:**
```sql
-- В postgresql.conf
max_connections = 200
shared_buffers = 4GB
effective_cache_size = 12GB
work_mem = 16MB
maintenance_work_mem = 1GB
```

#### 3.2. Кэширование (Redis)

**Что кэшировать:**
- Список задач (TTL: 5 минут)
- Статистика пользователей (TTL: 1 минута)
- Результаты тестов (TTL: 10 минут)

### Этап 4: Горизонтальное масштабирование

#### 4.1. Балансировщик нагрузки

**Варианты:**
- **Nginx** (бесплатно, просто)
- **HAProxy** (профессиональный)
- **Cloud Load Balancer** (AWS ALB, GCP LB)

**Конфигурация Nginx:**
```nginx
upstream flask_app {
    least_conn;
    server 127.0.0.1:5000;
    server 127.0.0.1:5001;
    server 127.0.0.1:5002;
    server 127.0.0.1:5003;
}

server {
    listen 80;
    location / {
        proxy_pass http://flask_app;
    }
}
```

#### 4.2. Множественные инстансы Flask

**Gunicorn с несколькими воркерами:**
```bash
gunicorn -w 4 -b 0.0.0.0:5000 --timeout 120 main_postgres:app
```

**Количество воркеров:**
- Формула: `(2 × CPU_cores) + 1`
- Для 4 ядер: 9 воркеров

### Этап 5: Мониторинг и логирование

#### 5.1. Мониторинг

**Инструменты:**
- **Prometheus** + **Grafana** (метрики)
- **Sentry** (отслеживание ошибок)
- **ELK Stack** (логи)

**Что отслеживать:**
- Время выполнения запросов
- Использование памяти/CPU
- Количество активных задач
- Ошибки выполнения кода

- Диск: 200GB SSD каждый

- Master: 8 ядер, 32GB RAM, 1TB SSD
- Replicas: 4 ядра, 16GB RAM, 500GB SSD

**Сервер очередей (Redis Cluster):**
- 3 сервера: 4 ядра, 16GB RAM каждый

**Балансировщик:**
- Отдельный сервер или облачный LB

**Стоимость:** ~$1000-2000/месяц

## 🔒 Безопасность

### Критичные меры:

1. **Docker контейнеры** для изоляции кода
2. **Ограничение ресурсов** (CPU, память, время)
3. **Отключение сети** в контейнерах выполнения
4. **Read-only файловая система** в контейнерах
5. **Rate limiting** (ограничение запросов от одного IP)
6. **WAF** (Web Application Firewall) для защиты от атак

### Дополнительные меры:

1. **HTTPS** (Let's Encrypt бесплатно)
2. **CSRF защита** (Flask-WTF)
3. **SQL injection защита** (используйте параметризованные запросы - уже есть)
4. **XSS защита** (Jinja2 auto-escape - уже есть)

## ⚠️ Важные замечания

1. **Начните с безопасности** - Docker контейнеры обязательны
2. **Масштабируйте постепенно** - не нужно сразу покупать мощное железо

4. **Тестируйте нагрузку** - используйте Apache JMeter или Locust
